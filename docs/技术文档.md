## 1. 项目概述

### 1.1 项目定位

Kitty-Cloud 是一个**个人专属的语音 Agent 系统**，实现类似豆包 App 的语音通话体验。用户通过 iOS App 一键拨打"热线电话"，与 AI 助手进行自然对话。

### 1.2 核心能力

| 能力 | 说明 | 技术实现 |
|------|------|---------|
| 语音识别 | 用户说话 → 文本 | 火山引擎 ASR (WebSocket) |
| 智能对话 | 文本 → AI 回复 | LangGraph Agent + DeepSeek |
| 语音合成 | 文本 → 语音 | 火山引擎 TTS (HTTP) |
| 流式响应 | 边生成边传输 | SSE (Server-Sent Events) |
| 多模型切换 | 动态选择 LLM | models.json 多 Provider 配置 |
| 会话持久化 | 重启不丢会话 | SQLite Checkpoint |
| KV Cache 优化 | 降低成本和延迟 | 三区划分 + 三不变式 |

### 1.3 技术选型

| 模块 | 选型 | 理由 |
|------|------|------|
| Agent Runtime | LangGraph `StateGraph` | 成熟的 Graph API，状态机 + 检查点 |
| Checkpointer | `AsyncSqliteSaver` | 异步 SQLite，零运维，个人项目最优 |
| LLM 调用 | `httpx` 直连 | 无额外服务依赖，可精确控制字节顺序 |
| 主力模型 | DeepSeek 官方 API | 自动 Prefix Cache，命中率 80%+ |
| API 框架 | FastAPI | 异步原生，自动 OpenAPI 文档 |
| 流式协议 | SSE | iOS 原生兼容，无需 WebSocket |
| 部署 | Host systemd + venv | 直接具备宿主机操作能力，程序与数据分离 |
| 数据库 | SQLite | 零运维，单文件备份，WAL 模式性能足够 |

## 4. Agent Runtime（LangGraph）

### 4.1 状态定义

```python
class AgentState(TypedDict, total=False):
    messages: Annotated[List[Any], add_messages]  # LangGraph add_messages reducer（追加而非覆盖）
    user_id: str                                  # 用户 ID
    session_id: str                               # 会话 ID（对应 LangGraph thread_id）
    model: str                                    # 用户指定的模型（如 "deepseek/deepseek-v4-flash"）
    recalled_memories: List[str]                  # 召回的长期记忆（Phase 2 预留）
    tool_results: List[Dict[str, Any]]            # 工具执行结果（Phase 3 预留）
    final_response: str                           # 最终回复文本
    llm_messages: List[Dict[str, Any]]            # build_context → agent 间的临时消息列表
```

**`llm_messages` 字段说明**：这是 `build_context` 节点的输出、`agent` 节点的输入，用于在节点间传递组装好的 LLM 消息列表。必须显式声明在 `AgentState` 中，否则 LangGraph 会静默丢弃该字段。

### 4.2 图结构

当前阶段（Phase 1）的图结构：

```
    START
      │
      ▼
┌──────────────┐
│ build_context │  ← 组装 system prompt + 历史 + 当前消息
└──────┬───────┘
       │
       ▼
┌──────────────┐
│    agent      │  ← 调用 LLM，生成回复
└──────┬───────┘
       │
       ▼
     END
```

**Phase 2/3 规划**（条件路由 + 工具循环）：

```
    START
      │
      ▼
┌──────────────┐
│ build_context │
└──────┬───────┘
       │
       ▼
┌──────────────┐     有 tool_calls
│    agent      │────────────────┐
└──────┬───────┘                 │
       │ 无 tool_calls            ▼
       │                  ┌──────────────┐
       │                  │     tools     │
       │                  └──────┬───────┘
       │                         │
       │                         └──→ 回到 agent
       ▼
┌──────────────┐

└──────┬───────┘
       │
       ▼
     END
```

### 4.3 节点实现

#### build_context 节点

**工厂函数**：`make_build_context_node(system_prompt: str)`

**职责**：
1. 从 `AgentState.messages` 取出 LangChain 消息列表
2. 调用 `build_llm_messages()` 组装最终 LLM 消息：
   - 转换为 OpenAI 格式
   - 按 Token Budget 裁剪历史（头部截断）
   - 按三区划分拼接消息
3. 将结果写入 `state["llm_messages"]`

#### agent 节点

**函数**：`async agent_node(state: AgentState) -> Dict[str, Any]`

**职责**：
1. 从 `state["llm_messages"]` 取出组装好的消息
2. 调用 `gateway.chat_sync()` 非流式调用 LLM
3. 返回 `{"messages": [AIMessage(...)], "final_response": reply_text}`

**注意**：此节点仅用于**非流式**路径。流式路径直接调用 `gateway.chat_stream()`，绕过图的执行。

### 4.4 两种执行路径

#### 非流式路径（`stream=false`）

```
run_chat()
  └── graph.ainvoke(input_state, config)
        ├── build_context_node → 写入 llm_messages
        └── agent_node → 调用 chat_sync → 返回完整回复
```

Checkpoint 由 LangGraph 在 `ainvoke` 时自动保存。

#### 流式路径（`stream=true`）

```
run_chat_stream()
  ├── build_llm_messages()        ← 手动调用 Context Engine
  ├── gateway.chat_stream()       ← httpx 流式调用 LLM
  │   └── yield {"type": "delta", "content": "..."}
  ├── yield {"type": "done", "metrics": {...}}
  └── graph.aupdate_state()       ← 保存 checkpoint（不重新调用 LLM）
```

**为什么流式路径绕过图？**

LangGraph 的 `astream_events` 可以捕获节点事件，但当前只有一个 agent 节点产生 token，直接用 `gateway.chat_stream()` 更简洁，延迟更低。图的主要价值在于状态管理和 checkpoint——流式结束后通过 `aupdate_state()` 手动保存即可。

### 4.5 Checkpoint 管理

**初始化**（`startup_agent()`）：

```python
from langgraph.checkpoint.sqlite.aio import AsyncSqliteSaver

_EXIT_STACK = AsyncExitStack()
_CHECKPOINTER = await _EXIT_STACK.enter_async_context(
    AsyncSqliteSaver.from_conn_string(checkpoint_db_path)
)
await _CHECKPOINTER.setup()
```

**关键细节**：
- 必须使用 `AsyncSqliteSaver.from_conn_string()` 而非直接传入 `aiosqlite.connect()` 的连接对象。直接构造会导致 `AttributeError: 'Connection' object has no attribute 'is_alive'`。
- 使用 `AsyncExitStack` 管理生命周期，确保关闭时正确释放连接。
- `from_conn_string` 内部创建懒连接，`setup()` 调用 `conn.is_alive()` 需要这个内部连接对象。

**存储位置**：

| 环境 | 路径 | 说明 |
|------|------|------|
| 本地开发 | `./data/checkpoints.sqlite` | 相对路径 |
| 生产部署 | `/home/yswwpp/dev/docker_file_sharing/kitty-server/data/checkpoints.sqlite` | 外部数据目录 |
| 环境变量 | `CHECKPOINT_DB_PATH` | 可覆盖默认路径 |

**清空会话**：删除 `checkpoints.sqlite*` 文件即可。

### 4.6 系统提示词

`VOICE_SYSTEM_PROMPT`（定义在 `agent/graph.py`）：

```
你是"Kitty"，一个温暖、活泼、善解人意的个人语音助手。

核心原则：
1. 短句优先——每次回复控制在 2-4 秒内说完
2. 口语化——自然对话，不用书面语或技术术语
3. 可被打断——用户随时可能插话，不要长篇大论
4. 不使用 Markdown 格式、不使用 emoji
5. 纯文本输出，直接说内容
```

**Cache 友好设计**：system prompt 不包含任何动态内容（当前时间、会话 ID 等），确保字节前缀完全稳定。

```
LLMGateway

**API Key 读取逻辑**：

api_key_env = provider_info.get("apiKeyEnv")  # 如 "DEEPSEEK_API_KEY"

```

### 5.3 模型解析

模型 ID 格式：`<provider>/<model>`

```python
# 示例
"deepseek/deepseek-v4-flash"  → ProviderConfig(name="deepseek"), "deepseek-v4-flash"
"litellm/gpt-5.5"             → ProviderConfig(name="litellm"),  "gpt-5.5"
```

**解析失败的情况**：

| 情况 | 报错信息 |
|------|---------|
| 未指定 model 且未配 DEFAULT_MODEL | `未指定 model，且未配置 DEFAULT_MODEL 环境变量` |
| model 格式不对（无 `/`） | `model 必须形如 'provider/model'` |
| provider 不存在 | `未知 provider: xxx，可用: [...]` |

### 5.4 非流式调用

```python
result = await gateway.chat_sync(
    messages=[...],              # OpenAI 格式消息列表
    model="deepseek/deepseek-v4-flash",
    temperature=0.7,             # 可选
    max_tokens=4096,             # 可选
    timeout=120.0,               # 默认 120 秒
)
# 返回: {"role": "assistant", "content": "...", "raw": {...}, "metrics": CacheMetrics}
```

### 5.5 流式调用

```python
async for chunk in gateway.chat_stream(messages=[...], model="deepseek/deepseek-v4-flash"):
    match chunk["type"]:
        case "delta":   # {"type": "delta", "content": "你"}
        case "done":    # {"type": "done", "metrics": {...}}
        case "error":   # {"type": "error", "message": "..."}
```

**流式实现细节**：
- 请求中携带 `"stream_options": {"include_usage": True}`，DeepSeek 在最后一个 SSE 帧中返回 `usage`
- 解析 SSE 行：`data: {json}\n\n`，遇到 `data: [DONE]` 结束
- 空行、非 `data:` 前缀行自动跳过
- JSON 解析失败静默跳过（不中断流）
- HTTP 错误（4xx/5xx）以 `{"type": "error"}` 返回

### 5.6 URL 拼接规则

```
base_url 已包含 /v1        → base_url + /chat/completions
base_url 包含 /v1/         → base_url + /chat/completions
base_url 不含 /v1          → base_url + /v1/chat/completions
```

| 限制 | 说明 |
|------|------|

## 6. Cache-First 调用策略

### 6.1 核心原则

> Cache stability isn't a feature you turn on; it's an invariant the loop is designed around.

DeepSeek 官方 API 支持自动前缀缓存（Prefix Cache），但匹配规则是**严格的字节前缀匹配**。大多数 Agent 实现每轮都会重排消息、改写内容或注入时间戳，导致实际缓存命中率极低。

本方案从架构层面保证字节前缀稳定，实测多轮对话 Cache 命中率 **80%+**。

### 6.2 上下文三区划分

```
┌─────────────────────────────────────────────┐
│  IMMUTABLE PREFIX（不可变前缀区）             │
│  - system prompt（固定，不含动态内容）         │
│  - 长期记忆（排序后拼接，顺序稳定）            │
│                                             │
│  特性：会话开始时计算一次，字节不变             │
│  → Cache 命中的核心保障                      │
├─────────────────────────────────────────────┤
│  APPEND-ONLY LOG（只追加日志区）              │
│  - user₁ → assistant₁ → user₂ → assistant₂  │
│                                             │
│  特性：只允许追加，不允许修改已有条目           │
│  → 上一轮的完整字节序在下一轮完整保留          │
│  → 仅最后一轮新增内容触发 cache miss           │
├─────────────────────────────────────────────┤
│  VOLATILE SCRATCH（挥发性草稿区）             │
│  - 推理链中间状态、临时计划                    │
│                                             │
│  特性：绝不发送给 LLM API                     │
│  → 需要持久化的信息经蒸馏后才进入 Log 区       │
└─────────────────────────────────────────────┘
```

### 6.3 三条 Cache 不变式

| # | 不变式 | 字节级影响 |
|---|--------|----------|
| 1 | **Prefix 计算一次、固定** | API 调用的前 N 字节在会话内完全一致 |
| 2 | **Log 仅追加、不重写** | 上一轮的完整字节序在下一轮完整保留 |
| 3 | **Scratch 经蒸馏后才入 Log** | 脏数据不会破坏 Log 区字节一致性 |

### 6.4 消息拼接实现

```python
def build_cache_stable_messages(
    system_prompt: str,
    memories: Optional[List[str]] = None,
    history: Optional[List[Dict[str, Any]]] = None,
    current_user_msg: Optional[str] = None,
) -> List[Dict[str, Any]]:
    """
    拼接顺序（一旦确定，整个会话期间不可改变）：
    [system] → [memories sorted] → [history...] → [current_user_msg]
    """
    messages = []

    # Zone 1: IMMUTABLE PREFIX
    messages.append({"role": "system", "content": system_prompt})
    for mem in sorted(memories or []):
        messages.append({"role": "system", "content": f"[记忆] {mem}"})

    # Zone 2: APPEND-ONLY LOG
    messages.extend(history or [])

    # Zone 3: Current user message
    if current_user_msg:
        messages.append({"role": "user", "content": current_user_msg})

    return messages
```

### 6.5 历史裁剪策略

当对话历史超过 `HISTORY_TOKEN_BUDGET`（默认 20000 字符）时，从**头部截断**：

```python
def trim_history(history, max_tokens):
    """Cache 友好的头部截断——删除最早的 user+assistant 对"""
    while estimate_tokens(history) > max_tokens and len(history) > 2:
        history = history[2:]
    return history
```

**为什么不从尾部截断？** 尾部是最近的消息，语义价值最高，且是上一轮 cache 命中的后缀。

**为什么不用摘要替换？** 替换会改变中间字节，导致从替换点往后全部 cache miss。

### 6.6 Cache 命中率统计

```python
@dataclass
class CacheMetrics:
    prompt_cache_hit_tokens: int = 0
    prompt_cache_miss_tokens: int = 0
    prompt_tokens: int = 0
    completion_tokens: int = 0

    @property
    def hit_rate(self) -> float:
        total = self.prompt_cache_hit_tokens + self.prompt_cache_miss_tokens
        return self.prompt_cache_hit_tokens / total if total > 0 else 0.0
```

DeepSeek API 响应的 `usage` 字段包含：
```json
{
  "prompt_cache_hit_tokens": 12345,
  "prompt_cache_miss_tokens": 678,
  "prompt_tokens": 13023,
  "completion_tokens": 256
}
```

**查看方式**：
- 服务端日志：每次 LLM 调用后输出 `[LLM] provider/model usage: ... hit_rate=81.53%`
- 健康检查：`GET /` 返回 `cache_stats` 字段

### 6.7 常见 Cache 杀手及规避

| Cache 杀手 | 典型做法 | 本方案规避 |
|-----------|---------|----------|
| System prompt 中注入当前时间 | `"现在是 2026-05-30 14:23"` | 时间信息通过工具获取，不写入 prefix |
| 每轮重排消息顺序 | 按 relevance 排序历史 | 严格按时间顺序追加，不重排 |
| 历史消息被改写/摘要替换 | 超长时压缩旧消息 | 只追加不修改；超长时从头部截断 |
| JSON 序列化顺序不一致 | dict 迭代顺序不确定 | 使用 `json.dumps(sort_keys=True)` |

## 7. Context Engine（上下文引擎）

### 7.1 职责

```
Context Engine (agent/context.py)
├── 消息格式转换（LangChain ↔ OpenAI）
├── 历史裁剪（Token Budget 管理）
├── 消息拼接（三区划分 + Cache 稳定）
└── 最终 Prompt 构建
```

### 7.2 Token Budget

```python
HISTORY_TOKEN_BUDGET = 20000  # 字符数（非精确 token，粗略估算）

# 估算规则（estimate_tokens）：
# - 中文字符：1 字符 ≈ 1 token
# - 英文单词：1 单词 ≈ 1 token
# - 每条消息：额外 +4 token（role 开销）
```

### 7.3 消息格式转换

```python
# LangChain → OpenAI（用于发送给 LLM API）
langchain_to_openai(messages: List[BaseMessage]) -> List[Dict[str, Any]]

# OpenAI → LangChain（用于将前端传来的历史注入 AgentState）
openai_to_langchain(messages: List[Dict[str, Any]]) -> List[BaseMessage]
```

### 7.4 最终组装流程

```python
def build_llm_messages(system_prompt, history_messages, recalled_memories=None):
    # 1. LangChain 消息 → OpenAI 格式
    history = langchain_to_openai(history_messages)
    # 2. 头部截断到 Token Budget 内
    history = trim_history(history, HISTORY_TOKEN_BUDGET)
    # 3. 三区划分拼接
    return build_cache_stable_messages(
        system_prompt=system_prompt,
        memories=recalled_memories,
        history=history,
        current_user_msg=None,  # 当前消息已包含在 history 末尾
    )
```

## 14. 设计决策记录

### 14.1 为什么用 LangGraph 而不是继续用 OpenClaw？

| 维度 | OpenClaw | LangGraph Agent |
|------|----------|----------------|
| 模型切换 | `x-openclaw-model` header | 直接 `model` 参数 |
| 流式响应 | 代理转发，多一跳延迟 | httpx 直连，延迟更低 |
| Cache 优化 | 无法控制 | 三区划分，命中率 80%+ |
| 调试排错 | 黑盒 | 全链路日志可控 |
| 部署依赖 | 额外进程 | 无外部依赖 |

### 14.2 为什么用 httpx 直连而不是 LiteLLM？

- 不引入新的服务依赖
- 延迟更低（少一跳网络）
- 可以精确控制消息拼接顺序，最大化 KV Cache 命中率
- 代码量很小（~300 行）

### 14.3 为什么用 SQLite 而不是 PostgreSQL？

Kitty 是个人助手，单用户场景，不存在分布式并发写入。SQLite 优势：
- 零运维：不需要额外进程
- 足够快：WAL 模式下单机读写性能优秀
- 备份简单：`cp checkpoints.sqlite` 即可
- 部署友好：单文件配置，支持外部路径

### 14.4 为什么不引入向量数据库？

Kitty 的长期记忆量级是"一个用户几年的偏好和对话摘要"，最多几千条。SQLite 的 LIKE 查询 + 关键词匹配完全够用。向量搜索需要额外部署 Qdrant/Milvus/Chroma，ROI 极低。

### 14.5 为什么流式路径绕过 LangGraph 图？

当前只有一个 agent 节点产生 token，直接用 `gateway.chat_stream()` 更简洁、延迟更低。图的主要价值在于状态管理和 checkpoint——流式结束后通过 `graph.aupdate_state()` 手动保存即可。Phase 3 加入工具调用后，可能需要重新评估是否走 `astream_events`。

### 14.6 为什么移除了硬编码的默认模型列表？

"默认认知"是不存在的——不存在对所有人都合理的默认模型。硬编码的 `bailian/qwen3.6-plus` 和 `bailian/glm-5` 在 API Key 未配置时完全不可用，却会静默地出现在 `/models` 响应中，误导用户和客户端。

**决策**：配置缺失就报错，宁可启动失败也不静默回退到不可用的默认值。

## 15. iOS 客户端语音功能

### 15.1 持续通话模式状态机

Kitty 采用持续通话模式，用户像打电话一样对话，保持持续连接。

```
状态流转：

idle ──▶ connected ──▶ listening ◀──────────────┐
                             │                     │
                             ▼                     │
                          thinking                 │
                             │                     │
                             ▼                     │
                          speaking ────────────────┘
                             │
                             ▼
                          idle（挂断）
```

**核心状态说明**：
- `idle`: 未通话状态
- `connected`: 通话已建立，音频引擎启动
- `listening`: 正在监听用户语音
- `thinking`: AI 正在处理和生成回复
- `speaking`: 正在播放 AI 回复（支持打断）

### 15.2 语音打断功能

用户在 Kitty 说话时可插话，立即停止 AI 语音，开始监听用户。

**实现参数**：
```swift
private let interruptionThreshold: Float = 0.15   // 音量阈值
private let interruptionDuration: Double = 0.3    // 持续时间（秒）
```

**逻辑**：在 `speaking` 状态下持续检测音量，超过阈值持续 0.3 秒即触发打断，立即停止播放进入 `listening`。

### 15.3 思考背景音乐

AI 思考过程中播放轻柔背景音乐，提升等待体验。

**音效特点**：
- 和弦进行：C大三 → A小三 → F大三 → G大三
- 12 秒循环，带琶音效果和颤音
- 进入 `thinking` 播放，离开时停止

### 15.4 耳机/蓝牙支持

动态检测蓝牙设备连接状态：
- **蓝牙模式**：使用 `.allowBluetooth` 选项，支持 HFP 协议（带麦克风）
- **无蓝牙**：使用 `.defaultToSpeaker` 选项，强制扬声器输出

**注意**：A2DP 协议只支持播放，HFP 协议才支持双向音频。

### 15.5 用户不活动检测

两种场景分别处理：
1. **用户说过后沉默**：5 秒后询问"还在听吗？"
2. **用户从未说话**：15 秒后询问"还在听吗？"
3. **60 秒无响应**：自动挂断

### 15.6 异常场景处理

| 场景 | 处理方式 |
|------|---------|
| 网络从 WiFi 切到蜂窝 | 无缝切换，不挂断 |
| 通话中接听电话 | 自动暂停对话，挂断后恢复 |
| 用户长时间不说话（>60秒） | AI 提示后 10 秒挂断 |
| AI 回复过慢（>3秒） | 播放缓冲音效"嗯…让我想想" |
| 用户多次打断 AI | AI 缩短回复长度 |
